HTTP

HTTP 协议也叫做超文本传输协议，是客户端和服务器端之间进行数据传输的格式规范，WEB 就是建立在 HTTP 协议上进行通信的

特性

  • 无状态协议

HTTP 是无状态协议，因为它不会对请求和响应之间的通信状态进行保存，协议对于请求和响应都不做持久化处理，这是为了更快的处理大量事务

  • 客户端可以主动向服务器发起请求，而反过来则不行

HTTP 协议规定，请求必须先从客户端发出，服务器才能响应请求并返回

  • 基于 TCP 实现

报文

用于 HTTP 协议交互的信息被称为报文，客户端的报文叫做请求报文，服务器端的报文叫响应报文

请求报文

请求报文由请求行、请求头、空行和请求数据四大部分组成

响应报文

响应报文由状态行、消息报头和响应正文组成

URL

URL 是一种特殊的 URI， 如果请求的 URL 协议，域名，端口部分不一样就会产生跨域

Request Method

请求方式有很多种，常用的就是 GET 和 POST

  • GET
    • 明文传输，参数包含在 URL 中
    • 传递数据长度有限制
    • GET 请求参数会保留在浏览器历史纪录中
    • GET 请求会被浏览器主动 Cache
    • GET 请求会将 http header 和 data 一起发送出去，只请求一次就把数据发送出去了
    • GET 参数通过 URL 传递
  • POST
    • 参数隐藏，不会显示在 URL 中
    • 传递数据长度可视为无限制
    • POST 请求参数不会保留在浏览器中
    • 浏览器不会 CachePOST 请求，除非手动设置
    • POST 请求会先发送 header，服务器响应 100 之后再发送 data，需要请求两次（Firefox 除外）

POST 请求相对 GET 虽然耗时，但是两次包的 TCP 在验证数据包完整性上有很大的优势

状态码

  • 1XX
    • 请求已接受，继续处理
  • 2XX
    • 请求已被成功接收、理解、接受
  • 3XX
    • 重定向，要完成请求必须进行更进一步的操作
  • 4XX
    • 请求有语法错误或请求无法实现
  • 5XX
    • 服务器未能实现合法的请求

常见状态码

  • 200 OK：客户端请求成功。
  • 400 Bad Request：客户端请求有语法错误，不能被服务器所理解。
  • 401 Unauthorized：请求未经授权，这个状态代码必须和 WWW-Authenticate 报头域一起使用。
  • 403 Forbidden：服务器收到请求，但是拒绝提供服务，比如对文件没有访问权限
  • 404 Not Found：请求资源不存在，举个例子：输入了错误的 URL。
  • 500 Internal Server Error：服务器发生不可预期的错误。
  • 503 Server Unavailable：服务器当前不能处理客户端的请求，一段时间后可能恢复正常，举个例子：HTTP/1.1 - 200 OK（CRLF）

HTTP1.0 和 HTTP1.1

  • HTTP1.0
    • 短暂连接，每一次请求都需要和服务器建立一个 TCP 连接，服务器处理完请求立即断开，不会记录过去的请求记录
    • 没有 host 字段，HTTP1.0 默认每一个服务器都绑定唯一的 IP 地址，所以请求的 URL 中没有包含主机名称
    • 缓存策略主要依赖于 header 中的 If-Modiified-Since,Expire
    • 存在一些性能浪费，请求对象中的一部分，但响应返回的是整个对象
  • HTTP1.1
    • 持续连接（长连接），同一个 TCP 连接中可以传送多个 HTTP 请求和响应，多个请求和响应可以同时进行
    • 存在 host 字段，HTTP1.1 的请求消息和响应消息都应支持 Host 头域，且请求消息中如果没有 Host 头域会报告一个 400 错误
    • 增加更多缓存策略
    • 加入更多错误状态响应码
    • 节约带宽，HTTP1.1 通过在请求头设置 range 头域，就可以返回请求资源的一部分，也就是返回码为 206 的时候，大大优化性能，如断点续传

缺陷

  • 无状态协议

无状态协议优点是可以快速处理大量事务，而缺点就是无法保存状态，举个例子：用户登录购物网站，即使跳转到该网站的其他页面也应该保持登录状态，但是无状态协议无法保存用户的登录状态，HTTP1.1 中加入了 cookie 的机制来缓存状态

  • 明文传输

HTTP 直接运行在 TCP 之上，所有传输都是用明文，明文传输可能会被窃听，即使已经加密过，也会被窥视到内容

  • 验证
  • HTTP 不会验证通信方的身份，因此有可能遭遇伪装
  • HTTP 无法证明报文的完整性，因此可能已遭篡改

URI 和 URL

URI

统一资源标志符 URI 就是在某一规则下能把一个资源独一无二地标识出来

URI 使用字符串来表示互联网中的某一资源，就像人的名字一样，可以通过人名知道这个人是谁，身份正也是一个很好的例子

URL

统一资源定位符 URL 是以描述位置来唯一确定资源的位置的

URL 描述的则是资源在互联网上所处的位置

无论如何 URL 都是 URI 的一种实现，因为它和 URI 一样通过一个方式确认一个资源的唯一性

HTTPS

HTTP 和 HTTPS 的关系可以理解为：HTTP + 加密 + 认证 + 完整性保护 = HTTPS

HTTPS 是一个披着 SSL 外壳的 HTTP，HTTP 直接使用 TCP 进行通信，而 HTTPS 先和 SSL 通信，再和 TCP 通信，SSL 是独立于 HTTP 的协议，除了 HTTP 协议其他协议也可以使用 SSL

HTTPS 传输数据

浏览器使用 HTTPS 传输数据的流程：

  • 客户端向服务器端请求 HTTPS 连接
  • 服务端返回网站证书（公钥）
  • 客户端产生随机的密钥
  • 客户端使用公钥对对称密钥加密
  • 客户端向服务器端发送加密后的对称密钥
  • 客户端通过对称密钥加密的密文通信

HTTP 和 HTTPS

HTTP 和 HTTPS 使用连接方式不同，默认端口也不一样，HTTP 是 80，HTTPS 是 443，HTTP工作于应用层，HTTPS工作于传输层

缺点

  • HTTPS 虽然更加安全，但性能较差，多次握手，导致页面加载时间延长近 50%
  • HTTPS 连接缓存不如 HTTP 高效，会增加数据开销和功耗
  • 申请 SSL 证书需要钱，功能越强大的证书费用越高
  • SSL 涉及到的安全算法会消耗 CPU 资源，对服务器资源消耗较大

TCP/IP

模型

TCP/IP 协议模型是 Internet 的核心协议

基于 TCP/IP 的参考模型将协议分为 4 个层次，下图表示 TCP/IP 模型与 OSI 模型各层的对照关系：

TCP/IP 模型

  • 应用层
    • 决定了向用户提供应用服务时通信的活动，HTTP 协议处于该层
  • 传输层
    • 对上层应用层提供处于网络连接中的两台计算机之间的数据传输
  • 网络层
    • 处理在网络上流动的数据包，数据包是网络传输的最小数据单位
  • 数据链路层（网络接口层）
    • 处理连接网络的硬件部分，包括控制操作系统，硬件设备驱动，网卡及光纤等物理可见部分，硬件上的范畴都在其作用范围之内

OSI 模型

  • 应用层
    • 为用户的应用程序（例如电子邮件、文件传输和终端仿真）提供网络服务
  • 表示层
    • 这一层的主要功能是定义数据格式及加密
  • 会话层
    • 他定义了如何开始、控制和结束一个会话，包括对多个双向会话的控制和管理
  • 传输层
    • 对上层应用层提供处于网络连接中的两台计算机之间的数据传输
  • 网络层
    • 负责将数据报的网络层分组从一台主机移动到另一台主机
  • 数据链路层（网络接口层）
    • 处理连接网络的硬件部分，包括控制操作系统，硬件设备驱动，网卡及光纤等物理可见部分，硬件上的范畴都在其作用范围之内
  • 物理层
    • 将报文中的一个一个比特的数据从一个节点移到下一个节点

通信传输流

使用 TCP/IP 在进行网络通信的时候，会通过分层顺序与对方进行通信，发送端从应用层往下走，而接收端从链路层往上走

三次握手四次挥手

三次握手

建立一次连接需要三次握手

  • 第一次握手：客户端给服务器发送一个 SYN 报文，客户端处于 SYN_SENT 状态
    • 客户端发送网络包，服务端收到了，说明客户端的发送能力、服务端的接收能力是正常的
  • 第二次握手：服务器收到 SYN 报文之后，会应答一个 SYN+ACK 报文，服务器处于 SYN_REVD 的状态
    • 服务端发包，客户端收到了，说明服务端的接收、发送能力，客户端的接收、发送能力是正常的。不过此时服务器并不能确认客户端的接收能力是否正常
  • 第三次握手：客户端收到 SYN+ACK 报文之后，会回应一个 ACK 报文，客户端处于 established 状态
    • 客户端发包，服务端收到了，客户端的接收、发送能力正常，服务器自己的发送、接收能力也正常
  • 服务器收到 ACK 报文之后，三次握手建立完成

其中 ACK 报文是用来应答的，SYN 报文是用来同步的

为什么需要三次握手？

  • 第一次握手，服务器端可以确认服务器接收和客户端发送能力没有问题
  • 第二次握手，客户端可以确认服务器端接收和发送与客户端的接收和发送能力是没问题，但是服务器端不知道客户端的接收能力有无问题，因为客户端没有再次发送信息给服务器
  • 第三次握手，服务器就可以确认客户端的接收能力没有问题
作用
  • 确认双方的接收能力、发送能力是否正常
  • 指定自己的初始化序列号，为后面的可靠传送做准备
  • 如果是 HTTPS 协议的话，三次握手这个过程，还会进行数字证书的验证以及加密密钥的生成

四次挥手

终止一次连接需要四次挥手

  • 第一次挥手：客户端发送一个 FIN 报文，报文中会指定一个序列号。此时客户端处于 FIN_WAIT1 状态
  • 第二次握手：服务端收到 FIN 之后，会发送 ACK 报文，且把客户端的序列号值 +1 作为 ACK 报文的序列号值，表明已经收到客户端的报文了，此时服务端处于 CLOSE_WAIT 状态
  • 第三次挥手：如果服务端也想断开连接了，和客户端的第一次挥手一样，发给 FIN 报文，且指定一个序列号。此时服务端处于 LAST_ACK 的状态
  • 第四次挥手：客户端收到 FIN 之后，一样发送一个 ACK 报文作为应答，且把服务端的序列号值 +1 作为自己 ACK 报文的序列号值，此时客户端处于 TIME_WAIT 状态

注意：TIME_WAIT 状态下，客户端发送 ACK 之后不直接关闭，而是要等一阵子才关闭，因为要确保服务器是否已经收到了我们的 ACK 报文，如果没有收到的话，服务器会重新发 FIN 报文给客户端，客户端再次收到 ACK 报文之后，就知道之前的 ACK 报文丢失了，然后再次发送 ACK 报文

为什么需要四次挥手？

  • 第一次挥手：客户端想终止连接，发送 FIN 报文给服务器
  • 第二次挥手：服务器虽然收到了 FIN，但很可能并不会立即关闭 SOCKER，所以只能先回复一个 ACK，表示已经收到 FIN
  • 第三次挥手：服务器所有报文接收完毕，就向客户端发送 FIN 表示可以结束了
  • 第四次挥手：客户端收到 FIN，才能结束连接
