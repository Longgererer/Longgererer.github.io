浏览器的回流与重绘

写PC端页面的时候，样式可以做得非常花哨，酷炫也不会引起卡顿，但在移动端上受限于机能问题，如果无节制地使用各种CSS动画会导致页面卡顿，这是浏览器的回流与重绘造成的

浏览器的回流与重绘是什么？
reflow：回流，比如说改变了某元素的宽高等会影响页面布局的属性，那么浏览器就会自动回流(重排)页面，所有受影响的元素都需要重新绘制。
repaint：重绘，比如说改变了元素的样式风格，但不会改变页面的布局，那浏览器就会发生重绘

在浏览器中，不管是用JS还是CSS渲染页面都不可避免地要进行以下几个处理：

也就是： JS/CSS => style(样式) => Layout(布局) => Paint(绘制) => Composite(合成)
此图源于Google 上的一篇文章

在讨论回流与重绘之前首先要知道：

  • 浏览器使用的布局方式是’流布局’方式
  • 浏览器将CSSDOM树(由CSS构建而成)和DOM树(由HTML构建而成)合并成为Render Tree(渲染树)
  • 渲染树只包含渲染网页所需的节点(不可见元素除外，包括通过 CSS 隐藏的元素)
  • 布局将计算每一个元素的精确位置和大小(不可见元素除外，包括通过 CSS 隐藏的元素)
  • 然后进行绘制，通过渲染树将像素渲染到屏幕上

渲染树是如何构建的

  1. 从 DOM 树的根节点开始遍历每个可见节点
  2. 对于每个可见节点，为其找到适配的 CSSOM 规则并应用它们。
  3. 发射可见节点，连同其内容和计算的样式。

此时，style(样式)阶段已经完成，浏览器已经计算了哪些节点应该是可见的以及它们的计算样式，在Layout(布局)阶段计算它们在设备视口内的确切位置和大小，浏览器会从渲染树的根节点开始进行遍历

现在进入正题

先说reflow

Reflow(回流)

Reflow是通过改变Layout触发的，前面说了，如果改变元素的样式会影响页面的布局，那么会导致浏览器重新检查所有元素，并且Reflow，所有受影响的元素都要重新Paint(绘制)，之后还要Composite(合成)

以下常用属性会导致浏览器Reflow：

  • width / height
  • margin / padding
  • border
  • dispaly
  • position
  • float
  • overflow / font-size / text-align / line-height …

除了样式之外：

  • 调整窗口大小
  • css 伪类
  • 输入框中键入文本
  • 计算 offsetWidth 和 offsetHeight
  • 一个操纵 DOM 的脚本
  • 操作类属性
  • 样式表发生改动
  • 在发生Reflow(回流)之后，一定会导致Repaint(重绘)

Repaint(重绘)

Repaint是当元素属性发生改变时触发的，除了导致Reflow(回流)的操作会导致Repaint(重绘)，比如修改背景图片、文字颜色、阴影等样式也会导致Repaint(重绘)，但不同的是，如果Layout(布局)没有发生改变，浏览器会直接跳过Reflow(回流)执行Repaint(重绘)，当然Composite(合成)是必不可少的。

  • 发生Repaint(重绘)不一定就会Reflow(回流)

也就是说Reflow(回流)比Repaint(重绘)的代价更高

如何避免 Reflow 和 Repaint

  • 减少更改设置样式的元素的类(尽可能地在 DOM 树的末端改变元素样式)
  • 内联样式尽量避免设置
  • 避免将动画应用于绝对或者固定定位的元素
  • 避免表格布局
  • 避免在 CSS 中使用 JS 表达式(例如:background:url(javascript:alert("test!"));)
