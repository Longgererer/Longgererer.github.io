Redux 的作用就是管理页面状态的，类似于 Vuex，是 flux 的一种实现

Action

Action 是把数据从应用传到 store 的有效载荷

Action 中包含的是我们发送到 store 中的数据，或者触发 store 中的数据更新，而且在 redux 中 action 中是 store 中数据更新的唯一来源。

添加一个新的 action

Action 本质上是 javascript 的普通对象，这个对象的规则是：

  • 必须使用一个 type 字段来表示将要执行的动作,可以理解为 action 的名称
  • 除了 type 之外，其他结构完全由你自己决定

官方建议参考 flux 的标准：

  • 一个 JavaScript 基本对象。
  • 必须含有 type 属性。
  • 可以包含 error 属性。
  • 可以包含 payload 属性。
  • 可以包含 meta 属性。

Action 不能包含除了 type, payload, error 和 meta 之外的任何属性。

应该尽量减少在 Action 中传递的数据，如果只需要 action 中的一个属性的值，就不需要将整个对象传过去

Reducer

Reducers 指定了应用状态的变化如何响应 actions 并发送到 store 的，记住 actions 只是描述了有事情发生了这一事实，并没有描述应用如何更新 state

Store 收到 Action 以后，必须给出一个新的 State，这样 View 才会发生变化。这种 State 的计算过程就叫做 Reducer

Reducer 是一个函数，它接受 Action 和当前 State 作为参数，返回一个新的 State

Reducer 与 Array.prototype.reduce 里的回调函数属于相同的类型，这是 Reducer 名字的由来

reducer 接收到名为 ADD 的 action 之后就返回一个新的 state

在实际应用中 Reducer 是通过 dispatch 自动执行的，在生成 store 的时候将 reducer 传入：

注意：Reducer 是一个纯函数，也就是说，传入同样的参数，得到的输出也是同样的

为了保持 Reducer 的纯净，最好不要在 Reducer 内做如下操作：

  • 修改传入参数
  • 执行有副作用的操作，如 API 请求和路由跳转
  • 调用非纯函数，如 Date.now() 或 Math.random()

最好就是将 state 设为只读，无法改变属性，这样就只能生成一个新对象来替换它，如果不这样做，redux 不会执行 listening 方法，因为 redux 认为对象没有更新过

Redux 工作流程

这是 Redux 官网的图片，上面显示了基本的工作流程

首先，React 组件里面如果需要状态（state），会先去找 Action Creators 请求状态或者改变状态，当改变状态的时候 Action 中其实是没有的，要去请求 Store 仓库，Store 再去请求 Reducers，Reducers 负责找到对应的 state 返回给 Store，Store 再返回给组件

举个例子，假如你是一个嫖客（组件），嫖客去夜店肯定是先找老鸨（Action）而不是小姐姐，嫖客先跟老鸨说自己要找谁，然后老鸨去大厅（Store）里找，但是没有找到小姐姐，所以就要问主管（Reducer）小姐姐忙不忙，然后主管找到小姐姐返回大厅交给嫖客
